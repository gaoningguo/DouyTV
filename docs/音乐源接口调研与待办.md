### 2.3 ⚠️ 我们当前调用的 bug(`cyreneApi.ts`)

- **`/kuwo/search`(`:34`)和 `/kuwo/song?mid=`(`:186`)对这个 OmniParse 后端是 404** —— 源码里压根没有 `/kuwo/*` 路由。omni 模式选酷我平台会失败。要么删掉酷我、要么确认你的 OmniParse 是带 kuwo 的分支。
- **`/kugou/song` 我们传 `hash=&album_audio_id=`(`:188`),但源码只认 `emixsongid`** —— 酷狗直链解析参数对不上,也会失败。需改成传 `emixsongid`(搜索结果里的 `emixsongid` 字段),或确认分支差异。
- `/song` 我们只用了 `type=json` 且没取返回里的 `lyric/tlyric`,等于每首歌白白多请求一次歌词。

> OmniParse 的 cookie 走服务端本地文件(`cookie/`),我们前端不用管。`.cyrene` 配置文件解密我们已实现(`cyreneConfig.ts`),解出 `{name,url,apiKey}`。

## 3. UnblockNeteaseMusic(`server`)

### 3.1 我们已调用

**无。** 目前没有任何代码对接 UNM。

### 3.2 接口形态(⬜ 全部未用,源码确证)

入口 `app.js` 逐行核实:UNM 启动的是 **http/https 双端口 MITM 代理服务器**(`server.http`/`server.https` + `hook` 拦截 + 白/黑名单 + endpoint 替换),**没有任何 REST 路由**。对外编程面只有两个:

- **MITM 代理**(主用法):改 hosts/PAC 或设系统代理,拦截 `music.163.com` 的 `song/enhance/player/url` 返回,把灰曲链替换成其他平台直链。CLI:`-p 端口` `-o 音源顺序` `-e endpoint` `-u 上游代理` `-s 严格模式` `-t 认证token` `-c 大陆中转`。**不适合我们**(我们是独立客户端,不走系统代理拦截网易请求)。
- **库函数** `match(id, source[], data)`(`provider/match.js`)→ `{url,br,size,md5,source}`。要用必须自己包一层 HTTP 服务(Node 进程内 `require` 调用)。

### 3.3 各 provider 真实能力(`src/provider/*.js`,统一 `check(info)` 模式)

每个 provider 导出 `check(info)`:内部 `search(info)`(按 `歌名 - 歌手` 关键词搜上游)→ `select()`(匹配最佳)→ `single()`(取直链)。`info` 来自 `find.js`(网易 `song/detail` 拿到的 `{name,artists,album,keyword,duration}`)。

| 代号 | 平台 | 默认启用 | cookie/依赖 | 备注 |
|---|---|---|---|---|
| `qq` | QQ 音乐 | | `QQ_COOKIE`(必,uin+qm_keyst) | 必须 QQ 登录 |
| `kugou` | 酷狗 | ✅ | 无 | 关键词搜→hash 取链 |
| `kuwo` | 酷我 | | 无(`KUWO_COOKIE` 可选) | `search.kuwo.cn/r.s` 关键词搜 |

### 3.4 结论(对我们项目)

| 能用在哪 | 怎么用 |
|---|---|
| 网易灰曲兜底 | 自建 wrapper 调 `match()`,或 **直接用 lxserver/enhanced 的 `/song/url/match`(等价能力,免自建)** |

> UNM 的核心价值(网易灰曲换其他平台直链)被 **enhanced API 的 `/song/url/match`** 完全覆盖。除非要 `youtube`/`bilivideo` 这类 UNM 独有上游,否则优先用 enhanced 解灰,**不必自建 UNM wrapper**。若真要接,因 UNM 无 REST,得起一个 Node 服务把 `match()` 包成 HTTP,成本高于直接用 enhanced。

## 6. TuneHub V3(闭源商业服务 `https://tunehub.sayqz.com/api`,代理抓到完整文档)

> 定位:**高性能音乐解析中间件,credit 计费**。支持平台:**netease / qq / kuwo**(仅 3 个)。
> 鉴权:请求头 `X-API-Key: th_xxx`(除公开接口外都要)。base URL = `https://tunehub.sayqz.com/api`。

### 6.1 ⚠️ 我们当前调用是错的(必须修)

- **现状** `cyreneApi.ts:223`:`GET {base}/api/?type=url&source=netease&id=&br=` —— 代理实测真实服务返回 `{"code":404,"message":"Not Found"}`。**TuneHub V3 根本没有 `?type=url` 这个接口,我们的 tunehub 模式实际是坏的。**
- 真实播放接口见 6.2。

### 6.2 真实接口全集(代理 curl 逐个实测确证)

**A. 核心解析 `/v1/parse`(消耗积分,需 key)**

| 路由 | method | 请求体/参数 | 返回 |
|---|---|---|---|
| `/api/v1/parse` | POST(JSON) + 头 `X-API-Key` | `{platform:"netease\|qq\|kuwo", ids:"id1,id2", quality:"128k\|320k\|flac\|flac24bit"}` | `{code,data:{total,success_count,fail_count,cache_hit_count,cost,data:[{id,success,fromCache,info:{name,artist,album,duration},cover,url,actualQuality,wasDowngraded,fileSize,lyrics}]}}` |

- 实测:无 key → `{"code":401,"message":"Missing API Key"}`;假 key → `{"code":401,"message":"Invalid API Key"}`。**必须有效 key 才能取直链。**
- `ids` 批量逗号分隔;`quality` flac24bit 仅 netease/qq(kuwo 不支持)。返回一次拿全 url+cover+lyrics+元信息。

**B. 方法下发 `/v1/methods`(不消耗积分,公开免 key,实测可用)**

| 路由 | method | 实测返回 |
|---|---|---|
| `/api/v1/methods` | GET | `{data:{kuwo:[search,toplists,toplist,playlist], qq:[...], netease:[...]}}` 三平台各 4 方法 |
| `/api/v1/methods/:platform` | GET | `{data:{platform,availableMethods:[...]}}` |
| `/api/v1/methods/:platform/:function` | GET | 上游请求配置 `{type:"http",method,url,params/body(含{{变量}}),headers,transform(JS函数串)}` |

**实测拿到的真实上游配置**(客户端拿到后替换 `{{变量}}` 自己请求上游,再用 `transform` 解析):

- `netease/search` → `GET music.163.com/api/search/get/web`,params `{s:{{keyword}}, type:1, offset:{{((page||1)-1)*(limit||20)}}, limit:{{limit||20}}}`,transform 取 `result.songs[]`→`{id,name,artist,album}`。
- `netease/toplists` → `GET music.163.com/api/toplist`,transform 取 `list[]`→`{id,name,pic,updateFrequency}`。
- `netease/toplist` & `netease/playlist` → `GET music.163.com/api/playlist/detail`,params `{id:{{id}}, n:100000, s:8}`;playlist 额外返回 `info{name,pic,desc,author,playCount}`。
- `qq/search` → **POST** `u.y.qq.com/cgi-bin/musicu.fcg`,body 是 `music.search.SearchCgiService` 的 JSON,query=`{{keyword}}`,transform 取 `req.data.body.song.list[]`→`{id:mid,name,artist,album}`。
- ⚠️ `kuwo/*` 的 methods 列在清单里,但实测单独取配置**返回空**(上游配置可能缺失/超时)——kuwo 方法下发不可靠。

> 模式总结:**搜索/榜单/歌单 = 方法下发,不扣积分,但客户端要会 eval `transform` JS 串 + 自己跨域请求上游**(浏览器端有 CORS 风险,文档明说要服务端/代理转发);**取播放直链 = `/v1/parse`,扣积分,需 key**。

**错误码**:`0` 成功 / `-1` 通用 / `-2` 积分不足 / `401` Key 缺失/无效 / `403` 封禁/Key 禁用 / `404` 资源不存在 / `500` 服务器错误。

**管理后台**(SPA 有 login/keys/users/credits/logs 页,对应后端端点实测未直接命中,非音乐对接所需,略)。

### 6.3 我们要做的(D)

- **修 `cyreneApi.ts` tunehub 模式**(现在 `?type=url` 是坏的):
  - 播放:改 `POST {base}/api/v1/parse` + 头 `X-API-Key`(用 `source.token` 存 key),body `{platform,ids:song.id,quality}`,从 `data.data[0].url` 取链,顺带存 `cover/lyrics`。
  - 搜索:`tunehub` 模式现在搜索其实走的是 OmniParse 端点(`SEARCH_SPECS`),与 tunehub 无关。要真用 tunehub 搜索得走方法下发(`GET /v1/methods/:platform/search` → eval transform → 请求上游),复杂度高;**建议 tunehub 源只做播放解析,搜索仍由聚合里其他源承担**。
  - platform 仅 netease/qq/kuwo;quality 直接传 `128k/320k/flac/flac24bit`(现在错误地传 `br={quality}`)。
- **注意**:付费服务,需用户自备 `X-API-Key`,UI 导入 tunehub 源要能填 key(存 `source.token`)。kuwo 方法下发不可靠,优先 netease/qq。
